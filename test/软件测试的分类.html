测试基础 #

什么是软件测试？ #

各位小伙伴再刚刚接触软件测试的时候，从各种资料教材上看到的定义都是一些很复杂的解释，看完也不是很清楚，软件测试到底是什么？其实软件测试很简单，大家作为初学者，只需要知道，软件测试就是在软件开发的过程中，去负责测试被开发的软件的各种内容的行为。软件测试工程师就是负责测试的人。软件在开发的过程中，可能会出现各种问题，所以需要专业的测试人员去对软件进行测试，比如你们安装在手机上的各种APP，除了写代码的人外，还有很多测试人员付出了很多努力，所以才能够在你们的手机上正常的运行，不出现问题。让各个用户能够在使用软件的过程中觉得好用，不出现问题，就是测试人员工作的意义。

软件测试的定义 #

在当前的时代里，软件测试其实并不是一个独立的工作了，而是一个综合性的工作，会涉及整个软件项目的方方面面，而不是只针对软件本身了。所以软件测试如今已经被定义为了。。。

软件测试的发展历程 #

1、证明软件是正确的
2、证明软件是错误的
3、软件测试是在软件开发的过程中保障软件质量的一种手段。

软件测试最新的流行方向 #

大家通过当前的测试的招聘要求可以看出来，目前国内软件测试的招聘要求都越来越偏向于技术面了，所以也就导致了软件测试的发展大家越来越趋近于软件开发。从自动化测试，到性能测试的要求，一直到如今流行的测试开发岗位，都可以看出国内的测试发展方向整体的偏向于了基础侧，并且在以后这个趋势会越来越明显。 测试和开发的岗位越发的趋近，也会导致测试和开发的思维模式相差的越远越专注。当大家会的技能都差不多了，只能从思维上体现出差距了。

软件测试的流程 #

软件测试是什么？ #

预期结果和实际结果做对比的一个过程。这个对比的过程就是软件测试。

  • 预期结果 所谓的预期结果就是我们在操作这个软件之前，我们期望它表现的一个结果。
  • 实际结果 实际结果就是我们正式的去操作了这个软件后，软件真实的表现的一个结果。

比如，现在要求你对电视机遥控板的音量加减的功能进行测试。 在这个遥控板的音量加减的功能的预期结果就是 在按下音量+的时候，电视机的音量就会对应的变大。按下音量-的时候，电视机的音量就会对应的变小。 在我们知道了预期结果的情况下，我们这个时候去操作这个功能，观察操作后的实际的结果是不是和预期的结果是一致的。 如果结果一致，那么就说明这个功能没有问题。 如果结果不一致，那么就说明这个功能有问题。

所以不管是对什么东西进行测试，只要我们知道它的预期结果是什么，知道怎么去操作这个东西，那么我们就可以对它进行测试。

软件测试的分类 #

按测试方法进行分类 #

  • 黑盒测试 所谓的黑盒测试就是把我们测试对象看作是一个不透明的黑色的盒子。 我们不知道这个盒子内部的结果和组成，我们也不需要去关心这个盒子内部的结果和组成。 我们只需要根据这个盒子所暴露出来的功能，根据对它操作时候的输入的内容（操作过程）和输出的结果（实际结果），预期结果和实际结果对对比就可以了。
  • 白盒测试 所谓的白盒测试就是把测试对象看作是一个透明的盒子，我们可以清晰的知道这个盒子的内部结构和组成，然后我们直接去检查盒子内部的结构组成是否正确的方式，进行测试。所以白盒测试是要求我们必须能够看的懂内部的结果的。 这个事情放到软件上，那所谓的白盒测试就是通过直接对代码进行检查的方式去判断软件是否正确。
  • 灰盒测试 灰盒就是一个半透明的盒子，我们对内部的结果可以看到，但是不能全部看到。 灰盒测试就是介于黑盒和白盒之间的一种测试方式，也就是同时结果黑盒的过程和白盒测试的过程就是灰盒测试。 灰盒测试的方法在实际的项目种用的挺多的。我们在做接口测试的时候就会用到灰盒测试。

按测试内容进行分类 #

  • 功能测试 对软件的功能进行测试，比如软件的注册、登录、发布朋友圈、点赞、视频等等功能。
  • 性能测试 对软件的性能进行测试。软件的性能要分成两种，1是客户端的性能，2是服务端的性能，一般我们说性能测试的时候都是说的服务端的性能，除非单独强调了是客户端的性能。
    • 客户端的性能 启动的速度、数据记载的速度、功能响应的快慢，页面的流程性之类的就是客户端的性能。
    • 服务端的性能 最多可以容纳多少用户使用软件，最多可以支持多少用户同时操作某个功能，在各种极端的情况下，服务器的响应的时间，最多每秒可以处理多少个请求，最多每秒可以处理多少条sql语句等等。
  • 安全测试 对软件的安全进行测试。安全测试的技术手段和黑客的攻击的手段是一模一样的。我们自己攻击自己家的软件，检查有没有可以被破击成功的漏洞之类的。常见的安全测试的手段，sql注入、ddos攻击、跨域攻击、渗透测试。。。
  • UI界面测试 对软件的页面进行测试，检查页面是否美观的这个过程就是UI测试。在做UI测试的时候，要看公司有没有软件的设计图，如果有设计图，那么UI测试就是直接拿软件的页面和设计图进行对比，检查展示的效果是否一致就行了。如果没有设计图，那么这个UI测试的过程就是一个非常主观性的过程。由你自己去判断，这个页面的展示效果是否美观，好看。当我们自己去做UI测试的时候，我们可以从UI的风格是否统一、配色是否合理、有没有错别字等方面去进行测试。
  • 易用性测试 易用性测试又叫做用户体验测试，测试人员站在普通用户的角度去使用这个软件，发现有没有不好用的地方。这就是所谓的易用性测试。所以易用性测试也是一个非常主观性的测试的过程。不同的人对易用性的感受是不一样的。 在做易用性测试的时候，我们可以围绕功能的操作的步骤是否可以精简，软件使用的时候的学习成本是否可以降低等等方面去进行测试。
  • 兼容性测试 检查我们的软件在不同的平台上能不能正常的使用。
    • APP的兼容性 APP是在手机上运行的，所以APP的兼容性测试就是测试这个APP在不同的手机上能不能正常的使用。APP的兼容性测试在测试的时候Android和IOS的版本是分别进行测试的，不能一概而论，就比如微信，虽然Android的微信和IOS的微信看起来似乎是一个软件，其实实际上是两个做成了一样的软件。在对APP的兼容性进行测试的时候，因为Android系统的手机有很多种，所以在实际的操作的过程中，因为的成本的问题是无法全面覆盖的，所以一般我们都需要根据手机的分辨率、系统版本、品牌、屏幕形状等等因素综合考虑，选择出覆盖率性价比最高的手机进行测试。而IOS的就没这么复杂了，只需要考虑不同的IOS的版本就行了。有同学在这里考虑到了APP是否需要去测试平板上的兼容性的这个问题，关于这个APP是否需要去测试在平板上的兼容性或者其他的设备的兼容性比如电视机，需要根据实际的情况来，也就是说，你要看你们测试的这个APP有没有要求兼容性平板，要求兼容电视机等等设备，要求了才去做，没要求就不管。
    • WEB的兼容性 WEB是在浏览器上面运行的，所以WEB的兼容性测试就是测试这个网闸在不同的浏览器上可不可以正常的使用。WEB的兼容性测试在做的时候，我们一般只需要考虑chrome、edge、firfox以这个三个浏览器为主进行测试，如果还有足够的时间和精力再考虑国内的浏览器，比如，QQ、360等。
  • 弱网测试 弱网测试就是测试我们的软件在不同的网络条件下能不能正常的使用。 不同的网络指的是2G/3G/4G/5G/WIFI。但是我们现在一般是不需要考虑2G和3G的情况的，毕竟实际情况下2/3G已经不是主流的网络了。 关于弱网测试的测试的手段就有2种，一种就是在实际的网络条件下进行测试，另一种就是模拟出对应的网络条件的对应的网速进行测试。
  • APP的专项测试 所谓专项测试是指的只有在测试APP的时候，才有的测试内容。
    • 安装、卸载、更新测试 检查App可不可以正常的安装、卸载、以及更新版本。在测试更新的时候，我们要考虑到低版本更新到高版本、高版本更新到低版本以及同级别更新三种情况。
    • 场景交互测试 APP在使用的时候的场景是十分的复杂的，所以我们要考虑到在不同的场景下，APP能不能正常的使用。比如，使用APP的时候电话来了，短信来了，一边听歌一边使用，小窗使用，分屏使用。不同的APP互相切换、桌面和APP互相反复切换等等。
    • 权限测试 APP要正常使用，是需要操作手机的各种权限的。所谓的权限测试，就是测试在关闭了对应的权限的时候，App的一个表现是否正常。比如一个APP，在使用拍照功能的时候，需要调用手机的相机权限，如果我们关闭了APP操作相机的权限，在这种情况下，APP和相机有关的所有的功能都无法正常使用，并且需要提示软件没有对应的权限，并引导用户去开启这个权限。
    • 消息推送测试 消息推送这个功能一般都不是直接利用APP的代码来实现的。消息推送一般会使用第三方的渠道来实现推送的过程，所以大多数的APP只要安装了，就算你没有启动也是可以收到APP的推送消息的。所以对于消息推送的这个功能我们就要测试在APP没启动的时候能不能收到、APP卸载后是不是就收不到了，发送了多条消息，是不是正常的按照时间顺序显示的，在没网的时候发送了推送消息，然后当手机连上网络后，可不可以收到。
    • 离线测试 APP有的功能，在没有网络的情况下也是可以正常使用的，所以我们就需要去测试在没网络的时候，会不会影响这些功能的使用，还有就是检查其他的功能在没网的时候，表现是不是正常的。因为有的APP做的不严谨，在没网的时候去操作，APP上面会爆出很多奇奇怪怪的错误提示。
    • 资源监测 资源监测也可以划分到APP的性能测试里面去。所谓资源监测就是看这个APP对你的手机的各种资源的消耗是否合理，比如硬盘、内存、CPU、电量、流量等等内容。 对硬盘的这个消耗很容易检查，就是看这个APP安装完成后所占用的硬盘的大小就可以了。其他的内容都是需要使用特殊的工具（adb）去进行监测的，或者使用手机自带的监测的功能也是可以实现的。

按测试的阶段分类 #

这里所说的阶段指的软件研发的过程的阶段。一个软件最开始的时候，不同的程序员每个人负责自己的开发的内容，大家各写各写的，互相干扰，这个时候程序员所写的代码我们就做单元。这里单元可以理解成开发写的函数、写的方法、或者写的某个功能的模块。 不同的程序员各自完成了自己的单元模块后，这个时候我们的软件是东一块西一块的，无法作为一个整体来运行。所以程序员需要把不同的单元模块集成起来，形成一个整体。开发做集成就是利用的软件的接口来做的集成。做完了集成后，那么不同的单元模块这些就成了一个整体，这个整体就是我们的系统了。

  • 单元测试 单元测试阶段就是对程序员所写的代码单元进行测试。在单元测试阶段所作的测试，需要使用白盒测试的手段来进行。不过也不需要担心，因为单元测试一般都是开发自己做。
  • 集成测试 集成测试就是对软件的接口进行测试。只要接口没有问题，那么集成的这个过程就不会出问题。 接口就是连接不同的模块之间的一个通道。通过接口把不同的模块给连接起来，然后就可以实现模块和模块之间互相通信。所以在集成测试阶段，我们主要做的测试工作就是对接口进行测试。
  • 系统测试 系统测试就是对整个完整的软件进行测试。在之间讲的这些测试内容，全部都是在系统测试阶段进行的。系统测试也是我们花费的时间和精力最多的一个阶段。
  • 验收测试 系统测试结束后，就代表这个软件的研发的任务完成了，就需要让甲方或者公司的领导来验收工作，来看看做出来的软件是不是符合他们的预期的。所以验收测试阶段和我们测试人员也没有太多的关系。在验收测试阶段我们就只需要辅助别人来进行验收就可以了。
    • α测试 领导或者甲方完成了验收测试后，但是依然不放心，担心还是存在问题，这个时候就可以组织公司的员工，作为一个普通的用户去试用一下这个软件，看看有没有什么问题。所以α测试又叫做内测。
    • β测试 完成了内测后，公司依然不放心，觉得可能公司的员工在试用的时候，有的功能无法像正常的用户那么去操作，所以公司又会组织更大规模的试用，选择一些正式的用户进行，先行下载试用，这个过程就是β测试，其实就是一个公测的一个过程。

按测试的手段分类 #

  • 手工测试 所有的测试的内容都考人工操作来进行。
  • 自动化测试 自己写代码，让你写的代码去帮你做测试，这就是自动化测试。

其他的测试分类 #

  • 全链路测试 全链路测试是接口测试里面的一个小的分类，全链路测试就是把一个系统的所有的接口根据业务流程全部串联起来运行，检查可以不可以跑通。
  • 打桩测试 打桩测试又叫做MOCK。mock是辅助程序员或者测试人员调试程序的一种手段。比如，某个前端的程序员已经完成了页面的开发了，现在打算开始去对接后端的接口，完成集成的这个事情。但是后端的接口还没做好。前端程序员又不想浪费时间等待后端完成接口的开发，这个时候，前端程序员就可以利用mock的手段，模拟出一些假的接口来试用。mock测试就是模糊出这个假的接口的这个手段。
  • 灰度测试 灰度测试就是在预生产环境上去检查软件有没有问题的一个过程。
  • 埋点测试 埋点测试时辅助开发或者测试工作的一种手段。有的时候我们需要收集一些用户操作的数据或者信息，我们就可以利用埋点的手段去收集。所谓埋点就是在用户操作的软件的代码里面，嵌入一些和功能无关的代码，用来收集用户操作的习惯和数据。这种代码就是埋点的代码，这周收集数据的手段就是埋点测试。对于我们测试来说，有的时候我们在测试时，测试的结果不是很直观的表现出来，我们就可以利用埋点的手段去收集这个测试的结果，然后直接去查看收集到的结果就可以很直观的知道了这个实际结果的表现了。
  • 冒烟测试 冒烟测试就是测试前的一个测试过程。目的是为了检查这个软件是否具备可测试性。冒烟测试只有在第一次测试之前才需要做的。比如你们公司突然安排你对某个软件进行测试，那么你首先就去做冒烟测试，检查这个软件是否具备可测试性。当这个软件都不需要测试，一眼就可以看到很多BUG，或者软件的核心的业务流程都无法走通，或者随便操作就会出现闪退、崩溃啊之类的现象，都可以认为冒烟测试没有测试通过，那么就是说这个软件不具备可测试性。也就是说你可以拒绝，不去测试，让开发这边把问题都改了再说测试的事情。
  • 回归测试 在日常的测试工作中，我们都会发现很多BUG，我们发现了BUG后我们是需要告诉开发，然后让开发去处理解决这些BUG的。所谓回归测试就是去检查开发有没有正确的修改在上一次测试中发现的BUG的一个过程。

思考 #

只要我们了解了什么是测试后，那其实不管是什么东西，我们都可以进行测试的。 那么在这里，假设给了你一个水杯、钢笔之类的东西，问你应该这么测试。 你们觉得应该怎么回答。

其实主要考察的就是测试的思路。任何东西都是一样的。

  1. 确认具体的需求是什么。
  2. 从测试的内容上面去思路，分门别类的列举出有哪些测试的内容。 2.1 功能测试 有哪些功能 2.2 安全测试 2.3 性能测试 2.4 UI测试 。。。
